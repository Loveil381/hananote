## 1. 研究目的

HanaNote v1.2.2 已上架并稳定运营。我们计划彻底重新设计前端（推翻当前「花笺」设计语言），但**在重做之前**，需要用真实用户数据替换团队主观直觉。

### 1.1 核心研究问题（Research Questions）

| RQ | 问题 | 用以决策 |
|----|------|---------|
| RQ1 | 当前 v1.2.2 哪些屏幕最常用、哪些最让人困惑？ | 重设计 21 屏的优先级排序 |
| RQ2 | 用户期望的"健康记录工具"应该是什么调性？ | 阶段 0 设计语言探索的方向（专业 / 温柔 / 中立 / 可爱）|
| RQ3 | 用户在用什么竞品或替代方案？为什么用 / 不用？ | 差异化机会与威胁识别 |
| RQ4 | 隐私 / 情感 / 数据三个价值的实际优先级权重？ | 重设计中三者的视觉权重分配 |
| RQ5 | Flutter Web (PWA) 与 Android 用户场景有何差异？ | 响应式设计断点与功能侧重 |
| RQ6 | 让用户犹豫继续使用的具体场景 / 故事？ | 关键流程的痛点修复 |

### 1.2 不在研究范围

- **不评估** PK 模拟器的医学准确性（属临床研究范畴，超出 UX 研究边界）
- **不收集** 用户实际 HRT 用药数据或生理指标（隐私第一原则）
- **不询问** 用户的医疗诊断细节（无医师资质，避免医疗建议口吻）
- **不做** 大样本量化研究（预算与时间不允许，留待发布后做问卷复检）

## 2. 研究方法

| 方法 | 用途 | 样本 | 时长 |
|------|------|------|------|
| **半结构化访谈** | RQ1/2/3/4/6 — 探索动机与态度 | 5–8 | 30 分钟/人 |
| **可用性测试** | RQ1/6 — 观察实际使用阻碍 | 同上（合并执行）| 15 分钟/人，含在 30 分钟内 |
| **竞品桌面扫描** | RQ3 补充 | N/A | 团队侧执行，1 天 |

**为什么选这两个方法：**
- 用户群小且分散（跨性别 + 三语言区），大样本问卷招募成本高、回收偏样严重
- 重设计早期需要"为什么"信号，定性方法对此最强
- 5–8 人即可饱和（参见 Nielsen 1993，定性研究 5 人即可发现 80%+ 可用性问题）

## 3. 招募

### 3.1 目标样本（n=5–8，目标 7）

按四个维度交叉覆盖：

| 维度 | 取值 | 至少各 1 人 |
|------|------|------------|
| **年龄段** | 16–22 / 23–28 / 29–35 | ✓ |
| **地区** | 中国大陆 / 日本 / 东南亚华人 | ✓ |
| **HRT 阶段** | 初始期（< 1 年）/ 稳定期（1–3 年）/ 长期（> 3 年） | ✓ |
| **使用平台** | 仅 Android / 仅 Web PWA / 两者都用 | ✓ |
| **HanaNote 使用时长** | 新用户（< 1 月）/ 中期（1–6 月）/ 长期（> 6 月） | 至少 2 名长期 |

> **说明：** 7 个样本，每人覆盖多维度。例如 P1 = 24 岁 / 上海 / 稳定期 / 仅 Android / 中期用户。

### 3.2 招募排除

- 18 岁以下（涉及未成年人健康数据，需监护人同意，超出本轮范畴）
- 有明显未公开身份顾虑导致访谈中难以保持自然交流者（招募时应明确说明可使用化名 + 视频可关，但若仍有强烈顾虑则建议改文字访谈或不参与）
- 与团队有直接利益关系者（同事、近亲）

### 3.3 招募渠道与文案

| 渠道 | 数量目标 | 备注 |
|------|---------|------|
| hrtyaku.com 社群（如有） | 2–3 | 主渠道，已是产品场景内用户 |
| Discord 跨性别相关 server（中文/日文/英文） | 1–2 | 注意 server 规则，先与 mod 沟通 |
| Telegram 群（华语 LGBTQ+） | 1–2 | 中国大陆 + 东南亚为主 |
| 私域（团队个人微信 / 朋友引荐） | 1–2 | 信任度高但样本偏 bias，不超过 30% |

**招募信文案（中文版）：**

```markdown
【HanaNote 用户访谈邀请】

你好！我是 HanaNote（一款 HRT 健康记录工具）的开发者。
我们正在重新设计应用界面，希望听到真实用户的想法。

参与方式：
- 30 分钟线上一对一聊天（视频或语音都可，全程可化名）
- 谈谈你日常如何记录身体变化 / 用药情况
- 看看现在的 HanaNote 哪里好用、哪里别扭

报酬：
- 30 元红包 / 价值等额的礼品卡（任选）
- 提前看到新设计稿（如果你愿意继续参与）

隐私保护：
- 全程录音仅用于内部分析，不外传，研究结束后 30 天内删除
- 你的身份信息（包括用户名）在我们的研究报告中会化名处理
- 任何问题你都可以选择不回答，随时可终止

如有兴趣，回复邮件 / 私信我，我会发简短的预筛问卷（3 题，1 分钟）。

—— [研究者姓名] / HanaNote 团队
```

**日文版与英文版**：保留同样结构，但翻译需要本土同行 review 调性是否合适。建议日文版加入更多敬语（です / ます形），英文版避免 "trans community" 这类标签化用词，使用 "people who use hormone therapy" 更中立。

### 3.4 预筛问卷（招募信回复后发送）

5 题，1 分钟内完成：

1. 你目前使用 HRT 多久？（< 6 月 / 6 月–1 年 / 1–3 年 / > 3 年 / 暂未开始）
2. 你用 HanaNote 多久？（< 1 月 / 1–6 月 / > 6 月）
3. 你主要在哪个平台用？（Android / Web 浏览器 / 两者都用）
4. 你常住地区？（中国大陆 / 日本 / 东南亚 / 其他）
5. 你愿意被录音吗？（仅音频 / 也可视频 / 不愿录音 — 仅笔记）

> 选择"不愿录音"的用户仍接受参与，但访谈员需更密集做笔记，单场可能需要拉长至 40 分钟。

### 3.5 同意书（Consent Form）要点

访谈前必须在线获得明确同意（语音确认 + 邮件留痕）：

- ✅ 录音仅用于内部分析，30 天后删除原始录音
- ✅ 转录文本中所有可识别信息（姓名、地点、社交账号）会被化名替换
- ✅ 研究报告引用的用户原话会以化名（P1 / P2…）呈现
- ✅ 用户可随时退出、可拒答任何问题
- ✅ 数据存储位置：本地加密硬盘（与 HanaNote 自身的本地加密哲学一致）
- ✅ 用户可在研究结束后申请查看自己的转录稿、要求删除任何片段

## 4. 访谈提纲（30 分钟，半结构化）

### 4.1 总体结构

```
00:00 - 00:05  暖场 (Warm-up)
00:05 - 00:10  使用上下文 (Context)
00:10 - 00:25  深入访谈 + 可用性任务 (Deep Dive)
00:25 - 00:28  反应与态度 (Reaction)
00:28 - 00:30  收尾 (Wrap-up)
```

### 4.2 暖场（5 分钟）

**目的：** 建立信任，让用户放松。**不录正式问题，仅聊天。**

- 自我介绍 + 强调"今天你的任何回答都没有对错"
- 确认录音同意 + 化名偏好（"今天我可以叫你什么？"）
- 闲聊 1 分钟（天气 / 此刻心情 / 今天有没有运动……）

### 4.3 使用上下文（5 分钟） — 对应 RQ3

> 目的：先了解用户更广的"健康记录"行为，HanaNote 只是其中一部分。

- Q1：你平时怎么记录身体的变化？除了 HanaNote 还用什么？
  - 探针：有没有用过别的 app / 用纸本日记 / 拍照存相册 / 不记录？
  - 探针：每种工具记什么、不记什么？为什么分开？
- Q2：你最早是怎么知道 HanaNote 的？
  - 探针：通过谁推荐 / 在哪看到 / 当时为什么决定下载？
- Q3：你用 HanaNote 之前在用什么？为什么换 / 加入 HanaNote？

### 4.4 深入访谈 + 可用性任务（15 分钟）— 对应 RQ1, RQ4, RQ5, RQ6

**先做 3 个可用性任务，再做 5 个目标问题。**

#### 4.4.1 可用性任务（10 分钟）— 屏幕共享操作

> 用户屏幕共享当前 v1.2.2，依次完成以下 5 个任务（实际仅做能完成的前 3–5 个，时间不够可截断）。**每个任务后立即问"刚才你是怎么想的？"**

| # | 任务 | 观察点 | 时长 |
|---|------|--------|------|
| T1 | 现在请假装是今天早上，你刚醒，需要服药。请用 HanaNote 记录你已服了今早的药。 | 是否能找到 Today；是否理解服药标记；标记后的反馈是否清晰 | 2 min |
| T2 | 现在请记录你今天测的一项身体数据（任意一项，比如体重 / 围度）。 | Record 入口可发现性；测量页 UI 友好度；保存的反馈 | 2 min |
| T3 | 假设你刚做完一次血液检查，请录入雌二醇（E2）的数值。 | Data 入口；血检录入流程；单位 / 日期等容易出错的地方 | 2 min |
| T4 | 请查看你过去 30 天的服药情况。 | Timeline 是否被发现；时间轴的可读性 | 2 min |
| T5 | 请把你的所有数据导出来，假设你要换设备。 | Profile / Settings 入口；导出按钮的可见性；用户对导出形式的预期 | 2 min |

**观察要素（访谈员现场记笔记）：**
- 完成时长 / 是否完成
- 求助次数 / 困惑表情
- 自言自语（thinking aloud 内容）
- 错误点击或回退操作
- 用户表达的情感（"这个还挺可爱的" / "啊？怎么是这样"）

#### 4.4.2 目标问题（5 分钟）

> 任务完成后立即问。如果时间已过，挑 Q4 + Q6 + Q8 优先。

- Q4 [RQ1]：刚才哪一步最让你卡住？
- Q5 [RQ1]：你日常用最多的是哪个 tab？基本不用的是哪个？为什么？
- Q6 [RQ4]：HanaNote 强调三件事 — 数据本地不上云的隐私、温柔的 UI 风格、详细的健康数据。如果只能保留两个，你会保留哪两个？
  - 探针：为什么放弃第三个？
- Q7 [RQ5，仅 Web 用户问]：你什么时候在浏览器用、什么时候在手机用？两者用法不一样吗？
- Q8 [RQ6]：你有没有过哪一刻想着"算了我不用 HanaNote 了"？发生了什么？

### 4.5 反应与态度（3 分钟）— 对应 RQ2

> 展示 4 张参考图（不是 HanaNote，是其他产品）作为调性候选。

**展示 4 张全屏截图：**
- A. 专业医疗类（如 MyChart 风格 — 蓝白冷调，数据密集）
- B. 温柔治愈类（如当前 HanaNote 花笺风 — 樱粉暖调）
- C. 中立工具类（如 Apple Health 风 — 系统中性）
- D. 二次元可爱类（如月经追踪 app Clue 的早期版本 — 圆角插画）

**问：**
- Q9：哪一张最像你心目中"理想的 HRT 记录工具"？
- Q10：哪一张你最不想要？为什么？
- Q11：如果给你做一个新的 HanaNote，你会希望它更靠近哪一张？为什么？

> **不要让用户猜"团队想要哪个答案"。** 强调没有标准答案，他们的偏好就是数据。

### 4.6 收尾（2 分钟）

- Q12：还有什么是我没问到、但你想让我们知道的？
- Q13：你认识的人里有没有也在用 HanaNote 或类似工具的？我可以联系吗？（雪球抽样）
- 致谢 + 红包发放方式 + 后续是否愿意看新设计稿 review

## 5. Codebook（初版主题编码字典）

> 用于阶段 -1.3 综合分析。每条转录文本片段标记一个或多个 code。可在执行中迭代扩充。

### 5.1 行为类（Behavior）

- `BEH-routine` — 日常使用习惯（每天 / 每周打开几次）
- `BEH-trigger` — 打开 app 的触发点（早晨 / 服药提醒 / 想起来 / 测了血检）
- `BEH-paper` — 同时使用纸本记录的场景
- `BEH-photo` — 拍照对比身体变化的频次与方式
- `BEH-multi-tool` — 同时使用多个工具
- `BEH-skip` — 长期不用某 tab 的行为

### 5.2 情感类（Emotion）

- `EMO-positive` — 正面情绪（"好可爱"、"很温柔"、"挺安心"）
- `EMO-negative` — 负面情绪（"看不懂"、"很烦"、"有点冷"）
- `EMO-vulnerable` — 脆弱时刻（孤独感 / 焦虑 / 自我怀疑）
- `EMO-celebrate` — 成就感时刻（达到某 PK 浓度 / 体测达标）
- `EMO-stigma` — 与身份相关的微妙情绪（隐私顾虑 / 不想被人看到）

### 5.3 痛点类（Pain）

- `PAIN-discoverability` — 找不到功能
- `PAIN-flow` — 操作步骤太多 / 顺序混乱
- `PAIN-copy` — 文案不清楚 / 让人困惑
- `PAIN-visual` — 视觉问题（对比度 / 字号 / 颜色）
- `PAIN-data-loss` — 数据丢失或担心丢失
- `PAIN-platform` — 平台特定问题（Web 慢 / Android 推送）
- `PAIN-i18n` — 翻译错误 / 文化不适

### 5.4 价值类（Value）— 对应 RQ4

- `VAL-privacy` — 用户提到隐私 / 本地存储 / 不愿上云
- `VAL-emotion` — 用户提到温柔 / 治愈 / 陪伴感
- `VAL-data` — 用户提到数据准确 / 趋势 / 科学
- `VAL-utility` — 用户提到效率 / 简洁 / 不打扰

### 5.5 调性偏好类（Tone）— 对应 RQ2

- `TONE-pro` — 偏好专业可靠
- `TONE-warm` — 偏好温柔治愈
- `TONE-neutral` — 偏好中立工具
- `TONE-cute` — 偏好可爱二次元
- `TONE-mixed` — 偏好混合（注明哪两种）

### 5.6 竞品类（Competitor）— 对应 RQ3

- `CMP-name:<name>` — 提及的具体竞品名
- `CMP-switch-from` — 从某竞品转过来
- `CMP-switch-to` — 考虑转去某竞品（流失风险）
- `CMP-coexist` — 与某竞品并用

### 5.7 平台类（Platform）— 对应 RQ5

- `PLT-android-only` — 仅在 Android 用
- `PLT-web-only` — 仅在 Web 用
- `PLT-both` — 两者都用
- `PLT-context-shift` — 在不同平台做不同的事（如电脑录入、手机查看）

### 5.8 流失类（Churn）— 对应 RQ6

- `CHURN-considered` — 曾考虑放弃
- `CHURN-reason:<...>` — 具体原因
- `CHURN-saved` — 什么把他们留下来了

## 6. 执行 SOP

### 6.1 工具栈

| 用途 | 工具 | 备注 |
|------|------|------|
| 视频会议 | 腾讯会议 / Zoom / Google Meet | 用户选 — 优先用户最熟的 |
| 录音 | 系统自带录音 + 备份录音机 | 双轨防丢 |
| 转录 | 飞书妙记 / Otter.ai / Whisper（本地）| 中文优先飞书；隐私敏感者用本地 Whisper |
| 笔记 | Notion / Obsidian | 现场实时笔记 |
| 红包 / 报酬 | 微信红包 / PayPal / Amazon GC | 按用户地区选 |
| 录音存储 | 本地加密硬盘（VeraCrypt） | 不存云，与 HanaNote 哲学一致 |

### 6.2 单场访谈流程

```
T-30 min:  设备测试（摄像头、录音、屏幕共享、网络）
T-15 min:  打开 codebook + 笔记模板，准备访谈提纲
T-5 min:   微信 / 邮件确认链接
T+0:       开始（自我介绍 + 同意书确认）
T+5:       进入暖场
T+30:      结束 + 立即整理 5 行核心观察
T+60:      上传录音到加密存储 + 启动转录
T+24h:     完成转录 + 编辑（去除可识别信息）
T+48h:     初步编码（按 codebook）+ 同行 review 一致性
```

### 6.3 隐私保护检查清单（每场必走）

- [ ] 录音前明确征求同意，并告知用途与销毁时间
- [ ] 用户表达不想录的片段必须立即停止录音并标注
- [ ] 转录稿中所有可识别信息（人名、地名、ID、社交账号）替换为占位符
- [ ] 不记录用户的具体诊断 / 用药剂量等医疗细节（即使用户主动提及）
- [ ] 录音文件存储在加密分区，访问需密码
- [ ] 30 天后销毁原始录音，仅保留化名转录
- [ ] 研究结束后用户可申请查看 / 删除自己的转录

### 6.4 访谈员守则

- **不打断**：用户说话时，停留 3 秒再回应
- **不引导**：避免"你是不是觉得 X 不好用？"，问"你怎么看 X？"
- **不评价**：用户操作错误时不要说"那是错的"，而是"你刚才在想什么？"
- **不护短**：用户批评 HanaNote 时不解释 / 不辩护，记下来即可
- **不医疗**：用户问"我这个药剂量对吗"，回答"我不是医生，不能回答这个；我们的研究只关心 app 的体验"
- **保持中立性别表达**：如不确定用户偏好，用第二人称"你"，避开"她/他"

## 7. 时间预算

### 7.1 1 周可行性评估

**结论：** 1 周完成 5–7 场访谈**勉强可行**，但建议宽松到 1.5 周（10 工作日）。

| 任务 | 单场时长 | 5 场总计 | 7 场总计 |
|------|---------|---------|---------|
| 招募信发送 + 筛选 | — | 2 天（持续） | 2.5 天 |
| 单场访谈本身 | 30 min | 2.5 hr | 3.5 hr |
| 设备测试 + 缓冲 | 45 min | 3.75 hr | 5.25 hr |
| 转录 + 整理 | 2 hr | 10 hr | 14 hr |
| 编码（codebook） | 1 hr | 5 hr | 7 hr |
| **总投入** | — | **~3.5 工作日** | **~5 工作日** |

### 7.2 7 天计划（n=6 场访谈）

```
Day 1: 完善预筛问卷 + 招募信发出（hrtyaku 社群、Discord、Telegram 各 1 篇）
Day 2: 收回执 + 筛选 + 约时间 + 完成 1–2 场
Day 3: 完成 2 场访谈 + 转录 Day 2 的
Day 4: 完成 2 场访谈 + 转录 Day 3 的
Day 5: 完成 1–2 场访谈（备选 / 补漏）+ 全部转录完成
Day 6: 编码所有转录 + 同行 review 一致性
Day 7: 阶段 -1.3 综合分析（调用 design:research-synthesis 喂模板 H）
```

### 7.3 风险与缓冲

| 风险 | 概率 | 缓冲 |
|------|-----|------|
| 招募回执率低于预期 | 高 | 提前 5 天开始招募；多渠道并行；同步雪球抽样 |
| 用户最后一刻退出 | 中 | 招募 1.3 倍预期人数（目标 7 → 招 9 人） |
| 转录工具中文不准 | 中 | 飞书妙记 + 人工 30% 抽查 |
| 时区差异（日本 / 东南亚） | 低 | 周末或晚间排期，明确告知 |
